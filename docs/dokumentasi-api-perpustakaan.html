POST /api/peminjaman Buat Transaksi Peminjaman
Request Body
{
  "kode_peminjaman": "PMJ2025070701",
  "id_anggota": 3,
  "id_petugas": 2,
  "tanggal_pinjam": "2025-07-07",
  "tanggal_batas_kembali": "2025-07-14",
  "buku_ids": [1, 5, 12]
}
Response · 201
{
  "success": true,
  "message": "Peminjaman berhasil",
  "data": {
    "kode_peminjaman": "PMJ2025070701",
    "total_buku": 3,
    "status_peminjaman": "aktif",
    "detail_peminjamans": [
      {
        "id_buku": 1,
        "status_buku": "dipinjam"
      },
      ...
    ]
  }
}
  1. Memulai Database Transaction
  2. Melakukan validasi input buku_ids yang wajib berupa array ID buku valid.
  3. Sistem memanggil Buku::whereIn('id_buku', ...)->lockForUpdate() untuk menahan/mengunci row data buku (menghindari race condition jika ada transaksi berbarengan).
  4. Sistem memvalidasi ulang: apakah stok_tersedia > 0? Jika tidak, Rollback dikirim dan pesan "stok habis" di-return.
  5. Jika stok aman, memotong stok_tersedia sejumlah -1 untuk masing-masing buku.
  6. Record Peminjaman dan Detail Peminjaman disisipkan ke tabel lalu Commit database dilakukan.

Perhatian Concurrency: Sistem menerapkan lockForUpdate() pada relasi DetailPeminjaman untuk mencegah 1 buku dikembalikan 2x secara bersamaan (Double-click Issue). Setelah sukses, sistem mengembalikan status peminjaman induk menjadi "selesai" (jika semua bukunya telah kembali) dan menambahkan stok_tersedia buku sejumlah +1.

Audit Trail Terintegrasi: Setiap kali pembayaran dilakukan (meskipun partial/cicilan), sistem memotong sisa_denda. Jika sisa denda 0, status berubah jadi lunas. Semua perubahan status otomatis di-record di tabel riwayat_status_dendas. Semua dibungkus di dalam DB Transaction + Pessimistic Lock untuk mencegah mahasiswa terhitung bayar ganda.

Kode Status HTTP

200OK — permintaan berhasil dijalankan (GET, PUT, PATCH)
201Created — data berhasil dibuat atau disisipkan (POST)
401Unauthorized — tidak ada token yang dikirim atau token invalid
403Forbidden — role user tidak diizinkan mengakses modul ini
404Not Found — data yang direquest (ID) tidak ada di tabel
422Unprocessable — input form gagal divalidasi
500Server Error — kesalahan internal database / sintaks backend

Keamanan & Role

Bearer Token

Autentikasi menggunakan Laravel Sanctum. Tiap API call mewajibkan Header Authorization: Bearer <token>.

CheckRole Middleware

Terdapat Middleware kustom yang menyaring akses. Admin memiliki kontrol master data penuh, sedangkan Petugas terbatas pada fungsi operasional (Sirkulasi, Katalog, Member).

Pessimistic Locking

Transaksi sirkulasi dan denda diproteksi MySQL lockForUpdate(). Menghindari selisih stok (race condition) akibat request klik ganda yang bersamaan.

Atomic Transactions

Relasi majemuk (Sirkulasi → Buku → Detail) dibungkus DB::transaction; ALL-OR-NOTHING.
